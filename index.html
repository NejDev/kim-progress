Progresso geral

O backend avançou para uma base bem próxima de produto real: além de catálogo, cliente, pedido, estoque e segurança, agora também existem validações melhores, cadastro de cliente por DTO, histórico de eventos do pedido, cancelamento, cupom de desconto, preparação de Mercado Pago, CORS e variáveis de ambiente para produção.

Fase 1 — Backend Core

concluído
  • ✓Category ModuleCategorias e subcategorias.
  • ✓Product ModuleProdutos ativos, slug, categoria e variantes.
  • ✓Customer ModuleCliente com conta, guest, login próprio e cadastro público por DTO.
  • ✓Address ModuleEndereços por cliente, acessados de forma segura pelo cliente logado.
  • ✓Admin + JWTLogin admin, BCrypt, JWT, roles e rotas protegidas.
  • ✓Order + OrderItemPedido, itens, total calculado no backend e respostas completas.

Fase 2 — Pedido real

concluído parcial
  • ✓EstoqueValidação antes de adicionar item e baixa automática quando pedido vira PAGO.
  • ✓Transição de statusFluxo controlado: PENDENTE → PAGO → EM_SEPARACAO → ENVIADO → ENTREGUE, com bloqueio de transições inválidas.
  • ✓Cliente acompanha pedidosCliente logado vê apenas os próprios pedidos e detalhes.
  • ✓Histórico de eventosPedidos registram eventos como criação, mudança de status, pagamento aprovado e cancelamento.
  • ✓CancelamentoCliente e admin possuem rotas de cancelamento com regras de negócio.
  • →Teste ponta a pontaValidar o fluxo completo com pedido real, evento, cupom, cancelamento e pagamento.

Fase 3 — Segurança & Qualidade

bem avançado
  • ✓Roles no tokenJWT diferencia ADMIN e CUSTOMER dentro do Spring Security.
  • ✓Permissões separadasCliente acessa área de cliente; admin acessa área administrativa; acesso indevido retorna 403.
  • ✓Rotas sensíveis protegidasPedidos, dados de cliente e endereços não ficam mais expostos por ID público.
  • ✓Swagger AuthorizeSwagger possui autenticação com token JWT.
  • ✓Validação de requestBean Validation foi adicionada para DTOs importantes como cliente, cupom e aplicação de cupom.
  • ✓Erros padronizadosRotas removidas retornam 404 ou 405, evitando erro interno genérico.

Fase 4 — Features de e-commerce

concluído parcial

  • ✓Eventos do pedidoTimeline do pedido disponível para cliente e admin.
  • ✓Cancelamento de pedidoRotas de cancelamento para cliente e admin foram adicionadas.
  • →Mercado PagoCriação de preference e webhook foram preparados; falta validar com credenciais reais/sandbox.
  • •Upload de imagensPreparar produtos e personalizações para receber arquivos e imagens.

Linha do tempo

02/06

Fundação do projeto

Spring Boot, PostgreSQL, estrutura inicial e módulos base do catálogo.

03/06

Admin e autenticação

AdminUser, BCrypt, JWT, login administrativo e rotas protegidas.

04/06

Pedido real

Order, OrderItem, enum de status, total calculado, rota admin protegida e remoção do endpoint temporário.

08/06

Estoque, status e pedidos completos

Validação de estoque antes de adicionar itens, baixa automática ao pagar, regras de transição de status e retorno de pedidos com itens.

09/06

Autenticação de cliente e pedidos próprios

Login de cliente com token CUSTOMER, rotas protegidas para acompanhamento de pedidos e bloqueio de acesso a pedidos de outros clientes.

10/06

Revisão de segurança e controle de acesso

Separação real entre ADMIN e CUSTOMER, Swagger Authorize, proteção de pedidos, clientes e endereços, e correção de erros 404/405.

11/06

Features avançadas de backend

Validações, DTO de cadastro de cliente, histórico de eventos do pedido, cancelamento, cupons, preparação do Mercado Pago, CORS e variáveis de ambiente.

Estratégia atual: validar o backend de ponta a ponta antes de iniciar o front-end, reduzindo retrabalho e deixando a API pronta para uma loja real.
